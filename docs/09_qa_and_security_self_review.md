# Step 9-10 実行結果: 自分のモックコードへのQA・セキュリティレビュー

[00_executive_summary.md](00_executive_summary.md) で明示した境界(実サイトへの脆弱性診断は行わない)は維持したまま、**自分が書いたモックのコード**に対しては実際にレビューと修正を行った。これは正当な権限内の作業であり、01章の課題1・2・6の「直した」根拠として機能する。

## 実施内容と結果

### 1. JSシンタックスチェック(全ファイル)

```
node --check app.js / article.js / listing-articles.js / listing-books.js
             / listing-career.js / product.js / search.js
```

結果: **7ファイルすべてOK**(実行日: 本セッション時点)。

### 2. XSS耐性レビュー → 実際に修正した

`grep -rn innerHTML mock/assets/js` で全箇所を洗い出した結果、`assets/js/app.js` の `jmedjCard()` (全ページ共通のカード描画関数)が、商品名・メタ情報を **文字列結合でHTMLに直接埋め込んでいた**(`el.innerHTML = '<h3>' + item.title + '</h3>' + ...`)。

現状は自分が用意した信頼済みの静的JSONのみを渡しているため実害はないが、将来これが実CMS・実DBのデータに置き換わった場合、タイトルや著者名に `<script>` 等が含まれていればそのまま実行される典型的なDOM-based XSSパターンである。

**修正内容**: `document.createElement` + `textContent` によるDOM構築に書き換えた(コミット差分参照)。`product.js` / `article.js` は元から `textContent` ベースで実装済みであり、今回 `app.js` を同じ方式に統一した。`grid.innerHTML = ""` のような「空文字代入によるクリア」は文字列を注入しないため問題なし。

**今後実装する際の必須ルール(03章 Step 8に追記すべき事項)**: ユーザー入力またはCMS由来の文字列をDOMに挿入する箇所は `innerHTML` を禁止し、`textContent` またはテンプレートエンジンの自動エスケープを必須にする。

### 3. i18nキー網羅チェック → 実行し、ギャップを可視化・一部修正

`data/i18n.ja.json` の21キーに対し、JS内での参照有無を機械的にチェックするスクリプト([tools/check_i18n_coverage.py](../tools/check_i18n_coverage.py)、リポジトリに同梱)を実行した。

初回結果: 21キー中10キーのみJSから参照(11キー未使用)。特に検索結果ページの「読み込み中」「0件時のタイトル・ヒント」「タブラベル」が **i18nファイルを定義したのに実際には読み込んでいなかった**(HTMLに直接日本語をハードコードしていた)。これは「文言を外部化する」という04章の方針が名目だけで実装に反映されていなかったことを意味する。

**修正内容**:
- `app.js` に `jmedjLoadI18n()` / `jmedjT(key, vars)` を追加。キーが存在しない場合は英語へ暗黙フォールバックさせず `[[key]]` の形でその場に表示する(未翻訳を隠さない設計。01章課題1の再発防止そのもの)。
- `search.js` を書き換え、読み込み中テキスト・0件時のタイトル/ヒント・タブラベル・結果件数表示を `jmedjT()` 経由に変更。
- `product.js` の「カートに入れる」ボタンを `jmedjT("cta.addToCart")` に変更。

再実行結果: **21キー中11キーが参照済み**に改善。残り10キー(検索フォームのプレースホルダー/ボタン、絞り込みクリア、並べ替え4種、CTA3種)は、HTML側に直接日本語で書かれており英語フォールバックのリスクはないため未接続のまま残している。**この判断自体もこのドキュメントで明示し、隠さない。**

### 4. アクセシビリティレビュー → 実際に修正した

`grep -c aria- *.html` で確認したところ、`account.html` / `article.html` / `product.html` に `aria-*` 属性が1つもなかった。個別に確認した結果:

- 見出し付きボタン(「カートに入れる」等)はテキストラベルがあるため実害なし
- **タブ切り替え(`検索結果ページのゾーンタブ`, `求人・物件ページの求人/物件タブ`)と、デモ状態切り替えボタン(`検索結果の3状態`, `アカウントページの未ログイン/ログイン済み`)は、視覚的な active クラスだけで状態を表現しており、スクリーンリーダー利用者には現在の選択状態が伝わらない** という実際の欠陥を発見

**修正内容**: 該当4箇所すべてに `role="tab"` / `aria-selected` または `aria-pressed` を追加し、クリック時のJSでも状態を同期するよう変更した(`search.js`, `listing-career.js`, `account.html`)。
