# 徹底批判: 現行サイトとモックv1を最新デザイン理論で斬る — そしてv2の設計原則

本章は2部構成である。第1部は現行 jmedj.co.jp を、第2部は**自分が作ったモックv1**を、同じ物差し(現代のデザイン理論・計測値)で容赦なく批判する。第3部で「1億MAU級」を目指すv2の設計原則を定義する。

批判は観測できた事実(2026-07-20取得のHTML、リポジトリのコード)にのみ基づく。未検証の推測は「推定」と明記する。

---

## 第1部: 現行 jmedj.co.jp への批判

### 1-1. パフォーマンス: 「973KBのHTML」は設計の放棄である

- トップページのHTML単体で973KB、検索結果ページで734KB(実測)。HTMLだけでこの重さは、CSS/JS/画像を含めた総ページ重量が数MBに達することを意味する。
- Googleの調査では、モバイルページの表示が1秒→3秒に伸びると直帰率は32%増加する。4G実効速度(数Mbps)でHTML 1MBのダウンロード+パースは、**LCP(最大コンテンツ描画)の悪化に直結**する。
- 734KBの検索結果ページの中身の大半は「実データの入らない空ループブロック約30回分」(01章)であり、**ユーザーに1バイトの価値も届けないペイロード**である。これはパフォーマンス予算(performance budget)という概念が設計プロセスに存在しないことの証明である。

### 1-2. ナビゲーション: 135リンクのドロワーは「ナビ」ではなく「ゴミ捨て場」

- ヒックの法則(選択肢の数に応じて意思決定時間は対数的に増える)に照らせば、135個のフラットな選択肢は**ユーザーに意思決定を放棄させる**設計である。
- Barry Schwartzの「選択のパラドックス」が示す通り、選択肢過多は満足度を下げ、行動自体を抑制する。医師という時間のないユーザー層に対して135択を提示するのは、ユーザー時間への敬意の欠如である。
- 個別記事・個別求人・個別物件へのディープリンクがメインナビに直接列挙されている。これは「ナビゲーション(場所の一覧)」と「フィード(新着の一覧)」という異なる情報構造の混同であり、IA(情報設計)の基礎的な誤りである。

### 1-3. 検索: コンテンツ企業として致命的な「検索の軽視」

- 記事・求人・物件だけでサイトマップ11分割(推定1万件超)のコンテンツを持つサイトは、**構造的に「検索ファースト」でなければならない**(Amazon・YouTube・Wikipediaと同じ分類のサイトである)。にもかかわらず:
  - 検索UI文言が英語のまま(「No results for」等) — ローカライズの完全性の欠如。日本語で診療の合間に検索する医師に英語のエラーを見せる。
  - SSRが空ループを返し、結果表示はJS実行に全面依存 — プログレッシブエンハンスメント(GOV.UKやBBCが実践する「コア機能はJSなしでも動く」原則)の真逆。
  - **検索結果画面のメインコンテンツ領域に外部イベントの広告カルーセルがサーバー出力されている**(実測: 検索結果コンテナ直後に `mainBanner-604782` カルーセル)。検索は「ユーザーが目的を明示した瞬間」であり、そこに無関係な広告を差し込むのは検索意図への裏切りであると同時に、遅延読み込みカルーセルはCLS(レイアウトシフト)の温床である。
  - 検索設定のJSON(ファセット定義・内部フィールド名)がHTMLに丸ごと埋め込まれ、クライアントに配信されている — 内部構造の不必要な露出とペイロードの無駄。

### 1-4. 情報の香り(Information Scent)の欠如

- Pirolli & Cardの情報採餌理論によれば、ユーザーは「目的に近づいている感覚(香り)」を頼りにリンクを辿る。現行サイトでは:
  - URLが `/blogs/product/product_28873` `/pages/corporate202660703` のような無意味な採番であり、URLから内容の香りが一切しない。「URLもUIである」(Nielsen)という原則の無視。
  - 記事1万件超に対し絞り込みが `?view=new/special/series` の3種のみ。診療科で探せない医学記事データベースは、**香りを辿る道が存在しない**。
- `jmedj.myshopify.com` という生ドメインへのリンク露出は、フィッシング詐欺と同じ「ドメイン不一致」パターンをユーザーに正規動線として提示している。医療者向けサイトの信頼設計として最悪である。

### 1-5. 一貫性と品質管理の不在

- ナビ内リンクテキストに `&nbsp;` や価格・著者・日付がベタ書きで混入し、リンクごとに情報量・書式がばらばら — コンテンツと表示の非分離。
- ニールセンの十原則の第4項「一貫性と標準」に対し、URL規則・文言言語・リンク書式のすべてで違反している。これは個々のバグではなく、**デザインシステムと品質ゲートが存在しない**という組織的問題の症状である。

## 第2部: 自作モックv1への批判(同じ刃で自分を斬る)

ユーザーの指摘は正しい。v1は「現行サイトの写経+化粧直し」に留まっており、以下の欠陥がある。

### 2-1. 【最重度】モバイルでナビゲーションが消滅する

- v1のCSSは `@media (max-width: 960px) { .zone-nav { display: none; } }` — **代替手段なしでナビを消しているだけ**。モバイルユーザーはトップページに来ても、検索以外にどこにも行けない。
- これは「レスポンシブ対応」ではなく「レスポンシブ隠蔽」である。モバイルトラフィックが過半を占める現代のWebで、この状態は現行サイトの135リンクドロワーより悪い(135リンクは少なくとも存在はする)。
- 批判した相手(現行サイト)より劣る部分を持ったまま「提案」を名乗った点は、v1の最大の欺瞞である。

### 2-2. リンク切れの山 — 「どこにリンクされているのやら」

- 会社情報メガメニューの全項目(会社概要・お知らせ・FAQ・プライバシーポリシー)、フッターのサポート列(FAQ・お問い合わせ・利用規約)、「広告掲載について」がすべて `href="#"` の死にリンク。
- サイトマップページ・FAQページ・お問い合わせページ・会社概要ページが**存在すらしない**。
- デッドエンド(行き止まり)はユーザーの信頼を最速で破壊する。1つの `#` リンクは「このサイトは未完成である」というメッセージであり、それが十数個あるv1は提案の体をなしていない。

### 2-3. 検索が「デモボタン」頼みで、体験として存在しない

- v1の検索結果画面は「読み込み中/結果あり/0件」をボタンで切り替えるだけの標本箱であり、**入力に反応するライブな検索体験がない**。
- サジェスト(入力補完)なし、入力中の絞り込みなし、タブに件数表示なし、スケルトンスクリーンなし。2026年の消費者向けサービスの検索として通用しない。

### 2-4. カテゴリとCTAの意味論が混乱している

- カードのバッジ(「紙」「記事」等)は**タップできない飾り**で、カテゴリで絞り込みたいユーザーの期待(ヤコブの法則: ECサイトの慣習ではタグはタップできる)を裏切る。
- 「すべて見る」「詳しく見る」という語彙が混在し、「その場で絞り込む(状態変化)」と「別ページへ移動する(場所変化)」の区別がUI言語として定義されていない。

### 2-5. その他

- 動画は「ゾーン」と定義したのに一覧ページが存在しない(ゾーン設計の自己矛盾)。
- スキップリンクなし、フォーカスリングのカスタム設計なし、タッチターゲット44px保証なし(WCAG 2.5.8 / Apple HIG)。
- ヒーローが標語のみで、検索ファーストと言いながら検索が従属的な位置にある。
- 視覚言語が「無難なカードUI」であり、100年の歴史を持つ医学出版社である必然性が皆無。どの企業のサイトにも見える。

## 第3部: v2の設計原則 — 「1億MAU級」に向けて

### 原則0: 誠実さの但し書き

MAUはデザインだけでは達成できない(コンテンツ供給・配信インフラ・SEO・アプリ展開・年月が必要)。ここで定義するのは「1億MAU級のサービスが共通して満たしている設計水準」であり、v2はそれを静的モックの範囲で実装する。

### 原則1: 検索ファーストIA(YouTube/Amazonモデル)

- ヒーロー=巨大な検索ボックス+人気キーワードチップ+4ゾーンショートカット。標語よりも入力欄。
- 全ページのヘッダーに検索を常設し、**入力するとサジェスト**(タイトル候補+最近の検索履歴)がキーボード操作込みで出る。
- 検索結果はタブに**件数を表示**し、入力のたびにライブ更新。0件時は代替キーワード提案つきの空状態。

### 原則2: モバイルはアプリとして設計する(サムゾーン理論)

- Steven Hooberの調査ではスマホ操作の約75%は親指1本。主要ナビは画面下部の**ボトムタブバー**(ホーム/さがす/出版物/記事/マイページ、各44px以上)に置く。
- ハンバーガーは補助とし、開けば**全ゾーンのアコーディオン+会社情報までの実リンク**を持つフルドロワー。
- モバイルファーストでCSSを書き、デスクトップを`min-width`で拡張する(v1は逆だった)。

### 原則3: 死にリンクゼロ・ポリシー

- サイト内のすべての `<a>` は実在するページに解決される。会社概要・FAQ・お問い合わせ・お知らせ・サイトマップ・カートをすべて実ページとして作る。
- 人間可読の**サイトマップページ**を用意し、フッターから常時到達可能にする(「どこに何があるか」への直接回答)。

### 原則4: UI言語の統一(状態変化 vs 場所変化)

| 操作 | UI | 語彙 |
|---|---|---|
| その場で絞り込む | チップ/チェックボックス/タブ | カテゴリ名そのもの |
| 結果の切替 | タブ(件数つき) | ゾーン名+件数 |
| 別ページへ移動 | テキストリンク | 「もっと見る →」に全サイト統一 |

- カードのカテゴリチップは**タップ可能**にし、該当ゾーンの一覧へ遷移させる。CTA(詳細へ)とは視覚・位置とも分離する。

### 原則5: 状態デザイン(読み込み・空・エラーは第一級市民)

- 読み込み=スケルトンスクリーン(スピナーより体感が速い: 知覚パフォーマンス研究)。
- 0件=次の行動を提案する空状態。カート空=おすすめ提示。これらを全一覧・検索・カートに実装。

### 原則6: ブランドの必然性 — 「百年の紙面 × 現代の検索」

- 100年超の医学週刊誌という資産を視覚言語に翻訳する: **和紙色の背景・明朝体ディスプレイ(Shippori Mincho B1)・朱色アクセント**という編集ヘリテージと、Noto Sans JPによる現代的UIの対比。
- ゾーンカラー(藍/深緑/紫/琥珀)は紙面のインデックスのように使い、どのゾーンにいるかを常に知覚させる。
- 「どの企業にも見えるカードUI」からの脱却。この見た目は日本医事新報社にしか成立しない。

### 原則7: アクセシビリティと入力品質(WCAG 2.2 AA志向)

- スキップリンク、`:focus-visible`リング、タッチターゲット44px以上、`prefers-reduced-motion`対応、ランドマーク/aria属性、キーボードで完結する検索サジェスト。

### 原則8: パフォーマンス予算の継続

- HTML each ≤ 30KB、CSS+JS合計 ≤ 100KB(圧縮前)を予算とし、実測を12章(QA)で継続する。空ループも広告注入も存在しない。

## 第4部: v3への改訂(ユーザーフィードバック反映)

v2公開後、ユーザーから2点の指摘を受けて改訂した。

### 4-1.「デザインがダサい」→ クリーン・メディカルへ全面刷新

- v2の「和紙×明朝×朱」は編集の格式を狙ったが、1億MAU級の消費者向けとしては古風・野暮ったい方向に振れていた。
- v3では **白ベース + メディカルティール(#0e8a86)+ ネイビー(#143a63)** のクリーンな医療系配色に刷新。紙テクスチャ・朱印モチーフ・明朝ディスプレイを廃止し、モダンな日本語ゴシック **Zen Kaku Gothic New**(数値・英字ラベルは Manrope)に変更。柔らかい影・大きめ角丸・広い余白・淡いグラデーションで、清潔さと信頼感を出した。

### 4-2.「ユーザーは"さがす"から入るのでは?」→ 検索を情報構造の頂点へ

これは情報設計の本質的な指摘だった。v2は「検索ファースト」と称しながら、ナビの序列は「出版物」が先頭で検索は従属していた。v3で以下を是正:

- **モバイルのボトムナビ中央に「さがす」を浮いたFAB(円形の主役ボタン)として配置**。親指が最も届く中央に、検索を最優先の行動として据えた。
- **デスクトップのグローバルナビの先頭に「さがす」** を追加(出版物より前)。
- **トップのヒーローの主役コピーを「まず、さがす。」に変更**し、カテゴリ(ゾーン)ショートカットは「カテゴリから探す」という検索の補助として明確に従属させた。

### v2/v3実装との対応表

| 原則 | 実装ファイル |
|---|---|
| 検索ファースト | `index.html`(検索ヒーロー), `assets/js/app.js`(サジェスト), `assets/js/search.js`(ライブ検索+件数タブ) |
| モバイル=アプリ | `assets/css/style.css`(モバイルファースト+ボトムナビ), `partials/header.html`(ドロワー) |
| 死にリンクゼロ | `company.html` `faq.html` `contact.html` `news.html` `sitemap.html` `cart.html` `videos.html` `video.html` 新設 |
| UI言語統一 | `app.js` のカード生成(タップ可能チップ+「もっと見る」統一) |
| 状態デザイン | スケルトンCSS、検索/一覧/カートの空状態 |
| ブランド必然性 | `style.css` デザイントークンv2(和紙・明朝・朱) |
| a11y | 全ページ共通partials+CSS |
| 予算 | `tools/build_pages.py`+QA計測 |
