# UI/UX再設計の意思決定根拠

`mock/` 配下の静的モックが実装している再設計方針と、01章のどの課題に対応しているかを対応付ける。

## 1. ヘッダー: メガメニュー化

- **変更前**: ハンバーガードロワーに135リンクがフラット格納(01章課題3)
- **変更後**: 常時表示のグローバルヘッダーに4ゾーン(出版物/記事/動画/求人・物件)+会社情報+検索+アカウントのみを配置。各ゾーム名にホバー/タップするとメガメニューが展開し、そのゾーン内の主要カテゴリ(最大6〜8項目)のみを表示する。個別記事・個別求人へのリンクはメガメニューに含めない。
- **根拠**: ミラーのロー(Miller's Law: 人が一度に把持できる選択肢は7±2程度)に基づき、第一階層の選択肢を7つ以内に抑える。個別コンテンツへの導線は「新着」「一覧」ページに委譲する。

## 2. 検索: 日本語完全ローカライズ + 意味のある空状態

- **変更前**: `noResultFor: "No results for "` 等の英語文言が残存(01章課題1)、SSRは空ループの繰り返し(01章課題2)
- **変更後**:
  - すべての検索UI文言を日本語リソース(`mock/data/i18n.ja.json`)から読み込む設計にし、英語がフォールバックとして露出しないことをモックで示す
  - 検索結果0件時は「"○○"に一致する結果は見つかりませんでした。表記ゆれ(全角/半角、診療科名の略称等)を変えて再検索してください」のような **意味のある日本語の空状態** を表示する
  - 検索結果はゾーン別タブ(出版物/記事/動画/求人・物件)で分け、混在した結果を一覧で流し込まない
- **根拠**: 空状態(empty state)はユーザーが離脱するかどうかの分岐点であり、「結果0件」を無言の空白ではなく次の行動を提示する画面として設計する。

## 3. パンくずリスト・URL表示の一貫性

- **変更前**: URL命名規則が複数並存(01章課題5)、myshopify.comドメインの露出(01章課題4)
- **変更後**: 全ページで `jmedj.co.jp/{ゾーン}/{カテゴリ}/{意味のあるslug}` の形式に統一し、パンくずリストのラベルとURLセグメントを一致させる。外部ドメインへの直接リンクは行わず、常に自ドメイン内の正規URLを経由する。

## 4. 商品・記事・求人・物件カードのデザイン統一

- **変更前**: ドロワー内でリンクテキストに `&nbsp;` や価格情報がベタ書きされ、リンクごとに情報量・書式がバラバラ
- **変更後**: 4ゾーン共通のカードコンポーネント(サムネイル・バッジ・タイトル・メタ情報・CTA)を定義し、ゾームごとにバッジ色のみ変える(出版物=紺、記事=緑、動画=紫、求人・物件=橙)。価格・日付・著者名などのメタ情報は構造化して表示し、テキストの中に埋め込まない。

## 5. パフォーマンス予算

- **変更前**: トップページ973KB、検索結果ページ734KB(いずれもHTMLのみ、01章)
- **変更後の目標値(モックでの設計目標として明記)**: 主要ページのHTML転送量を150KB以下、検索結果の初期表示(0件・ローディング状態含む)は空ループ等の無意味なマークアップを含めない。

## QA観点(03章 Step 9 に対応)

モックを実際の実装に落とす際は、以下を自動チェックに組み込むことを推奨する。

- UI文言の翻訳キー漏れ検出(未翻訳キーがビルドを失敗させる)
- Lighthouse CIによるページ重量・LCPの継続計測
- 主要フロー(検索→詳細→購入/応募、会員登録→ログイン)のE2Eテスト
