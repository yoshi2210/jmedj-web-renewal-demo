# 調査範囲の境界とセキュリティレビューへの導線

## この提案書がカバーしていないこと(明示)

本リポジトリの調査・提案は、**一般訪問者が通常のブラウザ操作で到達できる公開ページの構造分析のみ**を根拠にしている。以下は一切実施していない。

- 脆弱性スキャン・自動化ペネトレーションテスト
- ログイン試行・認証突破・権限昇格の試行
- 非公開エンドポイント・管理画面の探索
- レートリミットやWAFの検証を目的とした高頻度アクセス
- 個人情報・注文データ・会員データへのアクセス

理由: 対象(日本医事新報社 公式サイト)は実在の第三者企業が運営する本番稼働中のサービスである。チャット上での「自分が責任者である」という申告のみでは、侵入的なセキュリティテストを行うための正式なエンゲージメント範囲(対象環境・期間・許可されるテスト手法)を確認する手段がない。したがって、このリポジトリの分析はあくまで **公開UI/UXの外形的な観察** にとどめている。

## 01章で言及した技術的観察について

01章に記載した内容(Shopify基盤の利用、Algolia検索ウィジェットの導入、URL構造、サイトマップの分割数)は、いずれも `curl` で取得した公開HTML・公開XMLに含まれる情報であり、通常のWebブラウジングやSEOツールでも同様に観測可能な情報である。これらは脆弱性情報ではなく、UI/UX・パフォーマンス上の設計課題として扱っている。

## 今後、実際にセキュリティレビューを行う場合の推奨手順

正式にセキュリティレビュー(脆弱性診断・ペネトレーションテスト)を実施する場合は、[03_enterprise_website_build_process.md](03_enterprise_website_build_process.md) の Step 10 に沿って、以下を先に確定させることを推奨する。

1. **エンゲージメント範囲の文書化**: 対象ドメイン/サブドメイン、対象環境(本番 or ステージング)、除外対象(決済系・個人情報系など)を明文化する
2. **実施期間・実施者の特定**: 社内セキュリティチームか外部専門ベンダーか、実施担当者を特定し、対象サービス側の運用担当にも事前通知する
3. **本番環境への影響を避けるため、可能な限りステージング環境で実施する**
4. **発見した脆弱性の取り扱いルール**(責任ある開示、修正までの猶予期間、社内外への報告フォーマット)を事前に定める
5. Shopifyのようなマネージドプラットフォームを利用している場合、**プラットフォーム側の利用規約でペネトレーションテストの実施条件が定められていることが多い**ため、事前にShopify(または利用しているECプラットフォーム)側の規約を確認する

## このリポジトリの位置づけ

このリポジトリはあくまで「UI/UX・IA刷新の提案書」であり、セキュリティ監査報告書ではない。セキュリティ観点の意思決定(脆弱性の有無の判断等)には使用しないこと。
