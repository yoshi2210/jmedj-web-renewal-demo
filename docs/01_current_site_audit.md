# 現状サイト監査(証跡付き)

調査日: 2026-07-20
調査方法: `curl`によるパブリックHTML取得のみ(認証なし・自動スキャンなし)。取得したHTMLはこのリポジトリには含めない(著作権のある本文コンテンツを含むため)。本ドキュメントには構造分析に必要な最小限の抜粋のみを引用する。

## 0. 技術基盤の把握

トップページ・サイトマップのマークアップから、以下が確認できる。

- ECプラットフォームは **Shopify**(`/cdn/shop/t/60/assets/...`、`/collections/`、`/products/`、`/cart`、`/account/login` という典型的なShopifyルーティング)
- 記事コンテンツは Shopify の **Blog機能**(`/blogs/product`, `/blogs/movie`, `/blogs/dr9navi-recruit`, `/blogs/dr9navi-estate`)を4つの独立した"ブログ"として転用し、実質的に4つの異なる事業(記事・動画・求人・物件)を表現している
- 検索は **Algolia**(`algolia-search.css`, `algolia-search-header-pc.min.js`, `predictive-search.js`)を導入
- サイトマップは `sitemap.xml` → `sitemap_products_*.xml` / `sitemap_pages_*.xml` / `sitemap_collections_*.xml` / `sitemap_blogs_1.xml`〜`sitemap_blogs_11.xml` の分割構成。ブログ用サイトマップだけで11ファイルに分かれており、記事・求人・物件の掲載数が非常に多いことを示す。

## 1. 検索UIの多言語対応漏れ(重大)

検索結果ページ(`/search?q=...`)のHTML内に埋め込まれたAlgoliaウィジェットの文言定義を確認したところ、日本語サイトにもかかわらず**英語の文言がそのまま残っている**。

```
noResultFor: "No results for ",
noResultFound: "No result found",
oneResultFound: "1 result found",
resultsFound: "results found",
searchTitle: "Search result",
addToCart: "Add to cart",
allProducts: "See all products ",
clearFilters: "clearing the filters",
```

これらは検索結果が0件のとき・並べ替え/絞り込みUIを操作したときにエンドユーザーへ直接表示される可能性がある文言であり、サイト全体が日本語で統一されている中で明らかな未翻訳箇所として目立つ。ユーザーが指摘した「日本語とそれに対応するページが噛み合っていない」という印象の直接的な原因の一つ。

**影響度**: 高(全ユーザーが検索を使うたびに露出しうる)
**再現手順**: `/search?q=<任意のキーワード>` にアクセスし、該当ヒットが少ない/絞り込みUIを開く。

## 2. 検索結果ページのサーバーレンダリングが実質空(重大)

`/search?q=糖尿病` を取得すると、HTTPステータス200・**ダウンロードサイズ734,879バイト**が返る。しかし `.search-results` コンテナの中身を見ると、実データが一切入らない**空のループブロックが約30回繰り返し出力されている**(改行と空白のみのdiv/pの繰り返し)。実際の検索結果らしきマークアップ(`product-card` / `product-item` / `ais-Hits` 相当のクラス)はサーバー返却HTML中に**1件も存在しない**。

つまり実装は以下のようになっていると推測される:

- サーバー(Liquidテンプレート)側は、検索結果を空配列として扱いループを回している → 空ブロックが件数分(またはページング上限分)出力される
- 実際の検索結果は、ページ読み込み後にAlgolia InstantSearchが**別途JS実行で**DOMに描画する

この構成の問題点:

- **JS実行が遅い/失敗する環境(低速回線、JS無効、クローラ)では、検索結果が長時間または永久に空白のまま**表示される
- 意味のないマークアップが大量に転送されるため、検索結果ページだけで734KBという明らかに過大なペイロードになっている(同社トップページも973KBあり、同様の傾向)
- 検索結果ページはSEO上インデックスされないことが多いとはいえ、**初期表示の体感速度(LCP/CLS)を悪化させる**実装になっている

ユーザーが指摘した「検索の結果表示がいびつ」は、この空ループの大量出力と、その後にJSが結果を差し込むという二段構成のズレに起因すると考えられる。

**影響度**: 高(検索を使う全ユーザーの体感速度に影響)

## 3. グローバルナビゲーション(ドロワーメニュー)の情報過多・無階層化

トップページのハンバーガードロワー(`.drawer__nav`)を解析すると、**135件のリンクがフラットに** 格納されている。内訳を大分類すると:

| カテゴリ | 例 |
|---|---|
| お知らせ・重要通知 | 書籍回収のお詫び、価格改定のお知らせ、新サイトログインについて 等 |
| 出版物EC(紙・電子) | 新刊、近刊案内、改訂情報、正誤情報、jmedmookバックナンバー 等 |
| 記事(Web医事新報) | 私の治療、学術・連載、臨床一般、企画、プラタナス 等の個別記事へのリンク |
| 動画(医事新報チャンネル) | 個別動画へのリンク |
| 医師求人(dr9navi-recruit) | 個別求人へのリンク |
| 医療物件(dr9navi-estate) | 個別物件へのリンク |
| 会社情報・legal | 会社概要、沿革、FAQ、プライバシーポリシー、特定商取引法表示 等 |

これら性質の異なる7カテゴリ・**実質4つの独立事業**(出版物EC/記事メディア/動画メディア/求人・物件情報)が、単一のドロワーメニュー内に見出しレベルの視覚的階層をほぼ持たないまま並んでいる。個別記事・個別求人・個別物件へのディープリンクまでメインナビ内に列挙されており、「メニュー」というより「最新コンテンツの寄せ集め」に近い。

**影響度**: 中〜高(初見ユーザーが目的の情報にたどり着けない主要因)

## 4. ブランドドメインの露出漏れ

ドロワー内のお知らせリンクの一部が、正規ドメイン `www.jmedj.co.jp` ではなく **`https://jmedj.myshopify.com/pages/...`** という生のShopifyサブドメインを直接指している。

```
https://jmedj.myshopify.com/pages/corporate20250925_2  -> 新サイトでの初回ログインについて
https://jmedj.myshopify.com/pages/corporate20250925    -> サイトリニューアルのお知らせ
```

問題点:
- エンドユーザーがブラウザのアドレスバーで見慣れない `myshopify.com` を目にすることになり、フィッシングサイトと誤認されるリスクがある(医療従事者向けサイトとしては信頼性に関わる)
- `www.jmedj.co.jp` とドメインが異なるため、Cookie/セッション(ログイン状態)がそのまま引き継がれない可能性がある

**影響度**: 中(信頼性・ブランド一貫性の問題)

## 5. URL命名規則の不統一

観測されたURLパターンだけでも複数の命名規則が並存している。

| パターン | 例 | 問題点 |
|---|---|---|
| 手動連番+日付 | `/pages/corporate202660703` | `202660703` は日付として不正な文字列に見え、人が手入力で採番している痕跡。タイポのリスクが高い |
| 日付+枝番 | `/pages/corporate20251222_2` | 同上。意味のある一覧ページ(`/news/2026/07/`等)ではなく個別ページが乱立 |
| 意味を持たない採番ID | `/blogs/product/product_28873` | URLから内容が一切推測できず、共有・記憶に不向き |
| 求人ID | `/blogs/dr9navi-recruit/recruit_1085` | 同上 |
| 地域コード風ID | `/blogs/dr9navi-estate/nc_00142`, `/ka_00017` | プレフィックスの意味が非公開(都道府県コード推定だが規則が外部から読み取れない) |
| コレクション接頭辞方式 | `/collections/paper-new`, `/collections/ebook-new` | これ自体は比較的一貫しているが、他パターンと混在すると全体としての統一感がない |

**影響度**: 中(SEO・ユーザーの記憶/共有しやすさに影響)

## 6. コンテンツ量に対してナビ上の絞り込み手段が乏しい

`sitemap_blogs_1.xml`〜`_11.xml` という分割数から、記事・求人・物件の総数はShopifyの1ファイルあたり上限(通常1ファイル数千URL)を超える規模と推測される。一方でグローバルナビから記事一覧へアクセスする手段は `?view=new` / `?view=special` / `?view=series` の3種類のみで、診療科・カテゴリ別の絞り込み導線がメインナビに存在しない(記事本文中のタグとしてカテゴリ名は表示されるが、ナビゲーション要素になっていない)。

**影響度**: 中(コンテンツ量とナビゲーション手段のミスマッチ)
