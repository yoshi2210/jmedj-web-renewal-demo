# 設計思想・原則、そして「何がダメで、どう直したか」

## 1. 採用した設計原則

場当たり的な「なんとなく良さそうなデザイン」ではなく、以下の確立されたUX/エンジニアリング原則に基づいて意思決定した。各原則は「なぜその直し方を選んだか」の根拠であり、将来別の改修を検討する際も同じ物差しで判断できるようにするためにここに明記する。

| 原則 | 内容 | 本サイトでの適用箇所 |
|---|---|---|
| **ミラーの法則 / ヒックの法則**(選択肢が多いほど認知負荷と意思決定時間が増える) | 第一階層の選択肢を7±2以内に抑える | メガメニューを4ゾーン+会社情報の5項目に限定(index.html)。旧サイトの135リンクのフラット羅列を廃止 |
| **プログレッシブ・ディスクロージャー**(情報は必要になったタイミングで段階的に開示する) | ナビには「ゾーム」までしか出さず、個別コンテンツは一覧ページの絞り込みUIに委譲 | mega-panel内は最大8項目、個別記事・求人はナビに出さない |
| **ヤコブの法則**(ユーザーは他サイトで慣れたパターンを期待する) | EC的な「一覧+絞り込みサイドバー+詳細ページ」という、Amazon等で慣れたパターンを採用 | shop-books.html / articles.html / career.html の `.listing-layout` |
| **ニールセンの十原則 #1 システム状態の可視化** | 「読み込み中」「結果あり」「0件」を常に明示し、無言の空白を作らない | search-results.html の3状態設計 |
| **ニールセンの十原則 #9 エラーの認識・診断・回復の支援** | 0件時は「なぜ0件か」ではなく「次に何をすればいいか」を提示する | `search.noResultHint`(全角/半角、略称の違いを変えて再検索を促す) |
| **ニールセンの十原則 #4 一貫性と標準** | 色・余白・コンポーネントを共通トークン化し、ゾーンごとに崩さない | docs/07_design_system.md、全ページ共通の `style.css` |
| **情報の探索性(Information Scent, Pirolli & Card)** | ユーザーが目的の情報に「近づいている感覚」を持てるよう、カテゴリ・地域などのファセットで能動的に絞り込ませる | shop-books.html(ジャンル/形式)、articles.html(診療科/連載区分)、career.html(地域) |
| **パフォーマンス予算(Core Web Vitals的発想)** | ページ重量に上限を設定し、初期表示速度を設計段階から制約条件にする | docs/06のKPI目標(150KB以下)、docs/09の実測結果(最大約10KB) |
| **セキュア・バイ・デザイン(多層防御)** | 信頼できないデータを想定し、DOM挿入は常にエスケープされた経路を通す | `textContent`ベースのDOM構築(docs/09 Step2で実際に修正) |
| **ローカライズの完全性(サイレントフォールバック禁止)** | 未翻訳キーは英語に逃げず、欠落が可視化される設計にする | `jmedjT()` の `[[key]]` フォールバック(docs/09 Step3) |
| **WAI-ARIA Authoring Practices(タブ/トグルの状態通知)** | 視覚的な状態変化は、スクリーンリーダーにも同じ情報量で伝える | `role="tab"` / `aria-selected` / `aria-pressed`(docs/09 Step4で実際に修正) |

## 2. Before/After 対応表

「今何がダメで、どう直したか」を、01章で確認した課題ごとに、原因・修正内容・確認できる場所・適用した原則とセットで示す。

### 課題1: 検索UIが英語のまま(重大)

- **今何がダメか**: Algoliaウィジェットのデフォルト文言("No results for" 等)が未翻訳のまま本番に出ていた
- **なぜそうなったか(推定原因)**: サードパーティ検索ウィジェットの言語設定を、日本語ロケール用に上書きし忘れている。i18nの仕組み自体が「翻訳漏れを検知できない」設計になっている(漏れても英語にフォールバックするだけで誰も気づけない)
- **どう直したか**: `mock/data/i18n.ja.json` に文言を外部化し、`app.js` の `jmedjT(key)` を経由してのみ画面に出す。**キーが見つからない場合は英語に逃げず `[[key]]` という壊れた見た目で表示する**ことで、翻訳漏れが本番で"静かに"英語表示されることを構造的に防ぐ
- **確認できる場所**: `mock/search-results.html` を開き `?q=` に何も一致しない語(例: `zzz`)を入れて0件状態を見る。デモツールバーの「0件」ボタンでも再現可能。実装は `assets/js/app.js`(`jmedjLoadI18n`/`jmedjT`)と `assets/js/search.js`
- **適用した原則**: ローカライズの完全性、ニールセン原則(システムは利用者の言葉を話すべき)

### 課題2: 検索結果ページがSSR空ループの羅列(重大)

- **今何がダメか**: サーバーが返すHTMLに実データの入らない空ブロックが約30回繰り返され、734KBのうち大半が無意味なマークアップだった。JS実行が遅い/失敗する環境では長時間ほぼ空白
- **なぜそうなったか(推定原因)**: サーバー側テンプレートのループが「結果が0件のとき何を出すか」を設計しておらず、空配列をそのままループしている。結果の実描画をクライアントサイドJSに全面依存する設計と噛み合っていない
- **どう直したか**: 「読み込み中 / 結果あり / 0件」の3状態を明示的なUIコンポーネントとして設計し(`loading-row` / `card-grid` / `empty-state`)、いずれか1つだけが常に表示される状態機械にした。0件時も「意味のある1つのメッセージ」を返すだけで、無意味な空ブロックの繰り返しは発生しない
- **確認できる場所**: `mock/search-results.html` のデモツールバーで3状態を実際に切り替えられる。`assets/js/search.js` の `render()` 関数
- **適用した原則**: システム状態の可視化、エラー回復支援、パフォーマンス予算

### 課題3: ナビ(ドロワー)に135件がフラット格納(重大)

- **今何がダメか**: 出版物EC・記事・動画・求人・物件という性質の異なる4事業+個別コンテンツへのリンクが、階層のないまま1つのメニューに詰め込まれていた
- **なぜそうなったか(推定原因)**: 「ナビゲーション」と「新着一覧」の役割が分離されておらず、新しいコンテンツが増えるたびにナビへ直接追加され続けた結果、際限なく肥大化した
- **どう直したか**: 02章の4ゾーン構成に基づき、メガメニューの第一階層を「出版物/Web医事新報/動画/求人・物件/会社情報」の5項目に固定。個別コンテンツはナビから完全に除外し、各ゾーンの一覧ページ(`shop-books.html` / `articles.html` / `career.html`)に絞り込みUIとして委譲した
- **確認できる場所**: `mock/index.html` のヘッダーをクリック/タップしてメガメニューを開く。各パネルが8項目以内に収まっていることを確認できる
- **適用した原則**: ミラーの法則/ヒックの法則、プログレッシブ・ディスクロージャー

### 課題4: myshopify.comドメインの露出

- **今何がダメか**: 一部のお知らせリンクが正規ドメインでなく生の `jmedj.myshopify.com` を指しており、ブランド一貫性とログインセッション継続性を損なっていた
- **どう直したか**: モック内のリンクはすべて自ドメイン相当のパス(`/account`, `/shop/books` 等、02章のURL移行表)に統一し、外部の生ドメインへ直接遷移させる設計を排除した
- **確認できる場所**: `mock/account.html`(ログイン・マイページを`/account`配下に統合)、02章のURL移行表
- **適用した原則**: 一貫性と標準、信頼性(見慣れたドメインはフィッシング誤認を防ぐ)

### 課題5: URL命名規則の不統一

- **今何がダメか**: 手動連番+日付スラッグ、意味のない数値ID、地域コード風IDなど複数の命名規則が並存していた
- **どう直したか**: 商品・記事はいずれも `?id=` パラメータで一意に参照しつつ、実装時は02章の変換規則(`/shop/books/{意味のあるslug}` 等)に置き換える設計とし、パンくずリストのラベルとURLセグメントが対応するようにした
- **確認できる場所**: `mock/product.html?id=b3`、`mock/article.html?id=a3` のパンくず表示。02章URL移行表
- **適用した原則**: 予測可能性・学習しやすさ

### 課題6: 記事・求人・物件にカテゴリ絞り込みがない

- **今何がダメか**: 商品には`/collections/`という絞り込み単位があるのに、記事・求人・物件(推定でブログ系サイトマップ11分割=最も件数が多いコンテンツ群)には`?view=new/special/series`の3種類しか絞り込み手段がなかった
- **どう直したか**: 商品一覧と全く同じ`.listing-layout`(ファセットサイドバー+カードグリッド+0件時のempty-state)を articles.html(診療科・連載区分)と career.html(地域・求人/物件タブ)にも適用し、コンテンツ種別による絞り込み手段の非対称性を解消した
- **確認できる場所**: `mock/articles.html`、`mock/career.html` の左サイドバーでチェックボックスを操作すると件数がリアルタイムに変わる
- **適用した原則**: 情報の探索性、一貫性と標準

## 3. 実装過程で新たに見つけて直した問題(01章には無かったもの)

モックを実装しQA(docs/09)を実行する過程で、当初の監査には無かった以下の問題を**自分のコードに対して**発見し、その場で修正した。この経緯自体が「設計思想を後から検証する仕組みを持つことの価値」を示している。

1. カード描画共通関数がHTML文字列結合(`innerHTML`)でXSSパターンを内包していた → `textContent`ベースのDOM構築に修正(docs/09 Step2)
2. i18nリソースファイルを定義したのに実際には接続されていなかった(名目だけのローカライズ) → `jmedjLoadI18n`/`jmedjT`で実接続(docs/09 Step3)
3. タブ・デモ切り替えボタンにARIA状態がなくスクリーンリーダーに現在状態が伝わらなかった → `aria-selected`/`aria-pressed`を追加(docs/09 Step4)
